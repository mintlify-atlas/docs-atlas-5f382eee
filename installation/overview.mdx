## Which Method Should I Choose?

### Manual Installation

Best for:
- Existing SonarQube installations where you want to add branch support
- Environments where you manage SonarQube directly on servers
- Custom deployment scenarios

**Requirements:**
- Access to SonarQube installation directory
- Ability to modify `conf/sonar.properties`
- Ability to replace the `web` directory contents

### Docker

Best for:
- Quick setup and testing
- Container-based deployments
- Development environments

**Requirements:**
- Docker installed
- Ability to run containers

### Docker Compose

Best for:
- Complete local development setup
- Testing with a full database
- Quick proof-of-concept deployments

**Requirements:**
- Docker Compose installed
- Understanding of Docker networking

### Kubernetes

Best for:
- Production deployments on Kubernetes
- Organizations already using the official SonarQube Helm chart
- Scalable, cloud-native environments

**Requirements:**
- Kubernetes cluster access
- Helm 3 installed
- Knowledge of Kubernetes deployments

## Version Compatibility

<Warning>
  The plugin major and minor versions must match the SonarQube version. For example, version `25.4.0` of the plugin is compatible with SonarQube 25.4.x.
</Warning>

Always ensure you're using a compatible plugin version for your SonarQube installation. Using mismatched versions may result in errors or unexpected behavior.

## Important Notes

<Note>
  This plugin is not maintained or supported by SonarSource. Support is only available through GitHub issues or community channels like StackOverflow.
</Note>

- There is no official upgrade path to SonarQube Commercial Editions
- Migrating to a commercial edition after using this plugin may result in data loss
- Always follow the installation instructions for the specific version you're installing
